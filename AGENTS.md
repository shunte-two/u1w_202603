# AGENTS.md

## 目的

このプロジェクトは、ゲームジャム向けの 2D ゲームを短期間で立ち上げるための Unity テンプレートです。  
現時点ではテーマ未定のため、ジャンルを固定せず、既存の補助アセットを使って最短でプロトタイプを組むことを優先します。

## 作業開始前ルール

- 本作成を始める前に、必ず `AGENTS.md` を更新する
- 仕様変更、方針変更、使う補助アセットの変更があれば、その着手前に `AGENTS.md` へ反映する
- この更新を飛ばして作業を始めそうなら、先に `AGENTS.md` の更新をリマインドする

- 進行制御は `UniTask` を使った状態遷移で構成し、新規実装では `Coroutine` を使わない。

## 現在のフォルダ方針

- `Assets/Scenes`: 実プレイ用シーン
- `Assets/Scripts`: ゲーム固有スクリプト
- `Assets/Prefabs/Characters`: プレイヤーや NPC
- `Assets/Prefabs/Enemies`: 敵
- `Assets/Prefabs/Environment`: 地形、障害物、背景要素
- `Assets/Prefabs/Systems`: ゲーム管理、カメラ、スポナー、UI 以外の共通機能
- `Assets/Prefabs/UI`: HUD、メニュー、リザルト
- `Assets/ScriptableObjects/Databases`: バランス値や定義データ
- `Assets/ScriptableObjects/Stories`: 会話、演出、進行用の `StoryAsset` データ
- `Assets/Sprites`, `Assets/Animations`, `Assets/Materials`, `Assets/Audio` or `Assets/Sounds`: 素材格納

## 入力の前提

`Assets/InputSystem_Actions.inputactions` には、すでに以下の汎用アクションがあるため、テーマ決定前はこれを流用する。

- `Move`
- `Attack`
- `Interact`
- `Jump`
- `Sprint`
- `Crouch`
- `Previous`
- `Next`
- `UI` 系の標準操作

テーマ決定後も、最初の試作では入力アセットを増やしすぎない。  
不足が出るまでは既存アクション名に寄せる。

## ゲームジャム向け実装方針

- 最初の半日で「1 ループ遊べる状態」を作る
- テーマ解釈より先に、操作して気持ちいいコア体験を 1 つ決める
- ステージ数より、失敗と再挑戦のテンポを優先する
- システムは使い回せる粒度で作るが、過剰な抽象化はしない
- ScriptableObject は敵設定、アイテム設定、ステージ設定など再利用効果が高い箇所だけに使う
- 演出は DOTween で素早く付ける
- 非同期ロードや演出待ちは UniTask を使って複雑なコルーチンを増やしすぎない

## アセット参照と生成ルール

- `Resources` フォルダは新規作成しない
- `Resources.Load`、`Resources.LoadAsync` など `Resources` 経由の実行時ロードは使わない
- 実行時に `new GameObject` や `AddComponent` を組み合わせて GameObject をその場生成しない
- 実行時に生成が必要なオブジェクトは、必ず Prefab 化して Inspector から `SerializeField` で参照を受け取る
- 実行時生成は、Inspector で割り当てた Prefab を `Instantiate` して行う
- 文字列パスベースでアセットを引く実装より、Prefab 参照や ScriptableObject 参照を優先する
- `GameObject.Find`、`Transform.Find`、`FindFirstObjectByType`、`FindAnyObjectByType` などの探索参照は新規実装で使わない
- オブジェクト参照は Inspector からの `SerializeField` 割り当てを第一候補にし、シーン内 UI や管理オブジェクトもできるだけ明示参照で接続する
- 参照不足をコード内探索で補完しない。必要な参照があるなら、Prefab や Scene にシリアライズして持たせる
- 例外が必要なら、先に `AGENTS.md` に理由と許容範囲を追記してから着手する

## 推理ゲーム実装ガイド

### ゲーム概要

本作は、証拠カードの解釈を裏返すことで事件の矛盾を解消し、真相に近づいていく推理ゲームである。

プレイヤーは探偵として事件資料を調査し、提示された出来事を時系列で整理しながら、それぞれの出来事に対する解釈を切り替えていく。
重要なのは、カードに書かれている内容は事実そのものであり、変化するのはその事実の意味づけだけである、という点である。

コア体験は以下の 2 つで構成する。

- 同じ事象でも、見方によって意味が逆転する
- 出来事を正しい時系列と正しい解釈で並べたとき、事件全体が理解できる

### コアゲームループ

プレイヤーは以下を繰り返す。

1. 出来事カードを確認する
2. 各カードの解釈を表裏で切り替える
3. カードを時系列順に並べる
4. 現在の並びと解釈の組み合わせから矛盾が解消されているかを確認する
5. 条件を満たすと次の推理段階や物語進行が解放される

新しい証拠を大量に追加することよりも、既にある証拠の意味を読み替えることを主目的にする。

### コアギミック: 解釈反転

各出来事カードは以下の情報を持つ。

- 変化しない事実テキスト
- 表の解釈
- 裏の解釈

カードに書かれた事実自体は固定であり、プレイヤーはカードを裏返すことで別の解釈を選択できる。

表の解釈:
警察の推理、一般的で表面的かつ犯罪的に見える解釈。初期状態で表示される想定。

裏の解釈:
プレイヤーが見抜く別視点の意味。行動の意図や立場を反転させる解釈で、真相に近づく鍵となる。

反転対象は特に行動の解釈反転を重視する。

- 犯行か救助か
- 逃亡か追跡か
- 襲撃か警護か
- 証拠探しか証拠隠しか

行動自体ではなく、その行動の意図を反転させる。

### 出来事カード仕様

各カードは最低限以下を持つこと。

- `id`
- `factText`
- `frontInterpretation`
- `backInterpretation`
- `isFlipped`
- `correctTimelineOrder`
- `tags`

想定例:

- 事実: 被害者の後をつけていた
  表: 襲う機会をうかがっていた
  裏: 被害者を警護していた
- 事実: 被害者の家に忍び込んだ
  表: 犯行のために侵入した
  裏: 被害者を助けるために侵入した
- 事実: 被害者の机をあさっていた
  表: 物色して証拠を探していた
  裏: 危険な証拠を隠していた
- 事実: 現場から走り去った
  表: 犯人として逃亡した
  裏: 真犯人を追いかけた

### もう一つのコアギミック: 時系列再構成

プレイヤーは、出来事カードを正しい順番に並べることで、事件当夜に何が起きたかを再構成する。

各証拠は単体では意味が曖昧であり、以下の 2 つが揃ったときにのみ真相が見える。

- 正しい時系列
- 正しい解釈

単なるカード反転パズルではなく、順番と意味の両方を整える推理パズルとして設計すること。

進行判定では以下を確認できるようにする。

- カード順が正しいか
- 各カードの表裏状態が正しいか
- 必要に応じて、特定カード同士の因果関係が成立しているか

### 事件構造

ゲーム開始時点では、警察の推理が提示されている。

- 警察は全ての証拠を表面的に解釈している
- そのため、ある人物を犯人だと見なしている
- プレイヤーはその推理の矛盾を、証拠の読み替えによって暴いていく

プレイヤーの役割は以下である。

- 新証拠を大量に発見することではない
- 既存証拠の意味を読み替えること
- 時系列を整理すること
- 矛盾のない一貫した物語を再構築すること

### ストーリー上の前提

容疑者とされている人物は、真実を知っているが、それを警察に話せない事情がある。

- 彼が真相を語ると、犯罪組織に裏切り者と見なされる
- その結果、彼が守ろうとしている人物に危険が及ぶ

そのため彼は沈黙している。警察から見ると、その沈黙と不審な行動が犯人らしく見える。
しかしプレイヤーが行動の意味を順番に解き明かしていくことで、彼の行動が実は被害者を守るためのものだったと分かる。

### プレイヤー体験として重視すること

このゲームの面白さは、事実を増やすことではなく、同じ事実の見え方が反転することにある。

- 最初は明らかに犯人に見える
- だが、カードを裏返すと印象が変わる
- さらに時系列を正しく並べると、矛盾が消えて一本の筋になる
- 最終的に、犯人に見えた人物が実は守ろうとしていたと理解できる

特に重要なのは、すべてを裏返せば正解にならない設計にすること。
カードごとに、表が正しい場合も裏が正しい場合もありうる構成を望ましいものとする。

### 実装方針

最低限、以下の機能を実装すること。

- 出来事カードの一覧表示
- カードごとの表裏切り替え
- カードの並び替え
- 現在の並び順と解釈状態の保存
- 正解判定
- 正解時のストーリー進行

カード内容や正解条件は、できるだけデータとして定義すること。ロジックに直書きせず、章追加や事件追加がしやすい構造にする。

推奨データ:

- 事件データ
- カードデータ
- 判定条件データ
- 会話、演出データ

今後追加しやすい構造を前提とする。

- 複数事件
- 複数章構成
- 新しい解釈パターン
- ダミー選択肢
- 中間演出
- 一部カードのみ追加条件付きで反転可能

### UI/UX 上の注意

- カードの事実テキストは常に不変であると分かる見せ方にする
- 反転によって変わるのは注釈、意味、推理ラベルであると明確にする
- 表裏切り替えは直感的で気持ちよい演出にする
- 並び替え後に、時系列が変わった実感を持てるようにする
- 解釈変更によって矛盾が減ったことが分かるフィードバックを用意する

### NG 事項

以下は避けること。

- カードの事実文そのものが変わる実装
- 全カードを裏返すだけで正解になる単純設計
- 時系列要素が形だけで、順番が推理に影響しない構成
- 表解釈と裏解釈の差が弱く、反転の驚きがないデザイン
- プレイヤーが証拠を増やすゲームだと誤解する UI

### このゲームの一文要約

証拠カードの解釈を裏返し、時系列を再構成することで、犯人に見えた人物の行動が実は救助だったと見抜いていく推理ゲーム。

## Async Rule

- Unity Editor の確認、階層取得、スクリーンショット、ログ確認、コンパイル、テスト、PlayMode 操作など、Codex からの Unity 操作は今後 `uloop` を優先して使う
- Unity 用の MCP / 自動操作手段を選べる場合も、まず `uloop` のスキルと CLI を確認し、特別な理由がない限り `uloop` を第一候補にする

- 非同期処理は必ず UniTask を使う
- `Coroutine`、`IEnumerator`、`StartCoroutine` は新規実装で使わない
- フレーム待ち、時間待ち、`AsyncOperation` の待機も UniTask で統一する
